# Prerequisites

        This will take you some time to setup, but it is worth it!
        With this initial time invest you will save A LOT of time later.
        To ease management of all the required external entities an IaC tool
        like terraform is recommended.


To setup and use the kubara framework these are tools and external entities you need to prepare. Information from the external
entities should be added to .env to smoothen bootstrap process.

Here is an overview on how the .env looks like:
```.env
# These values MUST be known BEFORE running Terraform.
# Everything in <angle brackets> MUST be replaced.
# Dummy values (without <>) are optional and can be left as-is if not needed
#    (e.g. no private image registry). It will still create a secret, but it will be not valid.

### Project related values
### ownChange: prefix for all kubara related ressources
PROJECT_NAME='project-name'
### ownChange: naming prefix inside kubara
PROJECT_STAGE='dev'

### STACKIT Portal related values
# ownChange, (OPTIONAL) only necessary to use the `kubara generate --terraform` option
STACKIT_PROJECT_ID='0000000-0000-0000-0000-0000000000'

### Docker related values
# see https://docs.docker.com/reference/cli/docker/login/
# after successful login you can look inside envMap.json in your docker directory (~/.docker/envMap.json) on Linux/Mac
# the variable must be base64 encoded - how to: https://docs.kubara.io/latest-stable/6_reference/faq/#how-do-i-create-a-dockerconfigjson-for-env-file
# ownChange: (OPTIONAL), used to avoid the quota

DOCKERCONFIG_BASE64=''

### Argo CD related values
ARGOCD_WIZARD_ACCOUNT_PASSWORD='super-strong-password-here'

### Helm repository values
ARGOCD_HELM_REPO_USERNAME='example-user'
ARGOCD_HELM_REPO_PASSWORD='example-password'
ARGOCD_HELM_REPO_URL='example.com'

### Git repository values
ARGOCD_GIT_HTTPS_URL='git.example.com/dev-user/project-name.git'
ARGOCD_GIT_PAT_OR_PASSWORD='safe-argocd-password-or-token'
ARGOCD_GIT_USERNAME='git'

### DNS Name/Zones related values
# The Domain name under which your dns-entries will be added.
# The resulting dnsZone name will be a concatenation of <PROJECT_NAME>-<PROJECT_STAGE>.<DOMAIN_NAME>
# the value should be looking like 'stackit.zone' eg. 'yourDomain.com'
DOMAIN_NAME='example.com'

# Step 2: Helm and Argo CD Bootstrapping ...later with kubara --helm and kubara bootstrap..
# These values will only be known AFTER running Terraform (or manually via UI).
# They are usually available from Terraform output or Vault.

### Secrets-Manager/Vault for External Secrets Operator related values
SECRETS_MANAGER_USERNAME=''
SECRETS_MANAGER_USERNAME_BASE64=''
SECRETS_MANAGER_PASSWORD_BASE64=''
# 'SecretManagerPath' = Terraform Output: 'vault_instance_id'
SECRETS_MANAGER_PATH=''
SECRETS_MANAGER_URL='secretmanager.example.com'

### Currently only one additional vault for all workers is supported
# Sets the connection credentials for worker vault
WORKER_SECRETS_MANAGER_USERNAME_BASE64=''
WORKER_SECRETS_MANAGER_PASSWORD_BASE64=''

### This is only needed if you run on edge and use metalLB
### EDGE CLOUD related values
PRIVATE_LOADBALANCER_IP='0.0.0.0'
PUBLIC_LOADBALANCER_IP='0.0.0.0'
```

> **Important: Use 'single quoted' marks instead of "double quoted" to set variables**
Otherwise this might lead to env errors.
### Required Tools
- `kubectl`
- `helm`
- `git`
- `terraform` (optional, but recommended)
- `kubara` for your OS (Windows, Linux, or MacOS) - you can download it from the [releases page](https://kubara.git.onstackit.cloud/STACKIT/kubara/releases)
  - If you running on MacOS (Darwin) you can get some issues with unknown source. You can fix this by running:<br>
    ```bash
    xattr -d com.apple.quarantine ./kubara
    ```
    
> **⚠️ Note: Internet access required**
>   You will also need access to the internet to download the necessary Helm charts and CRDs.
>   Make sure you are not blocked by a firewall or proxy.


### External Entities
(necessary): You need to create these ressources yourself
(optional): You can create these ressources, but it's not required for kubara to run
(included\*): kubara can create some entities from scratch for you. At the moment we rely on STACKIT cloud services.

- **Git repository (necessary) (or multiple repositories)**  
  Used to store the code generated by the Kubara framework.  
  It is also used to configure and maintain your platform.  
  _Recommendation:_ Create one repository per control plane. Each repository should contain both the control plane configuration and all related worker clusters.
- **Kubernetes Cluster (included\*)**
A running Kubernetes Cluster (e.g. SKE or StackitEdge k3s with CNI) that can be used for running platform applications
- **Key Vault + Secrets (included\*)**
A place to securely store credentials (e.g. STACKIT Secrets Manager or Azure KeyVault).
See below which secrets should be inside vault
- **DNS Zone (included\*)**
A DNS Zone you can manage. ExternalDns will use this zone to request sub-domains for ingresses running on your cluster
enabling (public) access to your cluster
- **Helm Registry (optional)**
A Helm registry (e.g. JFrog or Harbor) that serves the (Umbrella-) Helm Charts for your platform
- **Container Registry (optional)**
A Container registry (e.g. JFrog or Harbor) that serves the applications
- **Docker Hub Account (optional but recommended)**
If you are relying on publicly available images (e.g. from dockerhub) you need an user account to pull without rate limits.
- **OAuth2 Apps (optional but recommended)**
    In total we use three Oauth2 Apps:
    -  OAuth2-Proxy App (reverse proxy) that handles authentication for most dashboards
    -  Argo CD App that handles authentication for Argo CD  UI. Dex is used for handling Argo CD  RBAC once authenticated
    -  Grafana App that handles authentication for Grafana.

---

After setting up these entities (or use existing ones) you need to provide some information to the kubara framework
(and store it in .env)

#### Infrastructure as Code (IaC)

We recommend provisioning key resources such as Kubernetes clusters, DNS zones, Vault keys and secrets, and OAuth2 provider configurations using Infrastructure as Code (IaC) - preferably with Terraform.

This ensures your platform setup is consistent, secure, and easily reproducible across environments.
